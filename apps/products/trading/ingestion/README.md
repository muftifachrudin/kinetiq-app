# Trading: Ingestion

Worker Python: connector CEX (CCXT+native WS) & DEX (Hyperliquid, dYdX v4, GMX, Vertex, Drift, Meteora DLMM, new-pair listener) -> normalizer -> fallback chain -> writer ke Neon. Lihat PRD Section B.11 utk rekomendasi sumber data.

## Status implementasi

**`connectors/cex/ccxt_generic.py` + `ingest.py`** — connector pertama yg beneran ada: **Binance USDS-M perpetual futures, Bybit, dan Hyperliquid** (DEX-perp pertama, Fase 1 B.9), `funding_rate` + `ohlcv` doang (scope sengaja sempit dulu, bukan strategy engine, bukan fallback chain native WS/Coinalyze; itu semua nunggu ronde berikutnya). `ccxt_generic.py` sengaja generik (bukan `binance_ccxt.py` per-exchange spt versi awal) — ccxt expose unified API yg sama persis di semua exchange, jadi satu wrapper cukup; nambah venue baru = tambah 1 entry di `VENUES` dict dalam `ingest.py`, asal exchange-nya didukung ccxt dgn `fetchFundingRate` (atau `fetchFundingRates`, lihat catatan Hyperliquid di bawah) + `fetchOHLCV`.

**Hyperliquid butuh 2 penyesuaian nyata di `ccxt_generic.py`, bukan cuma nambah entry**, ketauan lewat `exchange.has` ccxt langsung (bukan asumsi):
1. `fetchFundingRate` (single-symbol) itu `False` utk Hyperliquid — cuma ada `fetchFundingRates` (plural, all-market). `fetch_funding_rate()` di `ccxt_generic.py` sekarang cek `exchange.has.get("fetchFundingRate")` dulu, fallback ke `fetch_funding_rates([symbol])[symbol]` kalau falsy — struktur hasilnya tetap sama (unified `FundingRate` shape), jadi sisa kode gak perlu tau bedanya.
2. Auth Hyperliquid pakai `walletAddress`/`privateKey`, bukan `apiKey`/`secret` spt CEX — tapi gak relevan di sini krn `funding_rate`/`ohlcv` public endpoint di semua venue (gak butuh key sama sekali). `VENUES["hyperliquid"]` set `api_key_env`/`api_secret_env` ke `None`/`None` (bukan nunjuk ke env var yg bakal di-ignore diam-diam oleh ccxt), dan `make_exchange()` skip config kredensial sepenuhnya kalau salah satu `None`.

**Bug nyata ketemu & di-fix pas nambah Hyperliquid**: `FUNDING_INTERVAL_HOURS` sebelumnya di-hardcode `8` utk SEMUA venue — ternyata ccxt beneran expose field `interval` per-response (`"8h"`, `"1h"`, dst, sudah ada di Binance & Bybit juga, cuma gak pernah dipakai). Hyperliquid funding settle **per jam** (`1h`), jauh beda dari asumsi 8h — kalau tetap hardcode, data funding Hyperliquid bakal salah total. Sekarang `fetch_funding_rate()` parse `interval` dari response ccxt, fallback ke `DEFAULT_FUNDING_INTERVAL_HOURS = 8` cuma kalau venue itu bener2 gak expose field ini.

**Symbol convention beda**: Hyperliquid perp di-quote pakai USDC (`"BTC/USDC:USDC"`), bukan USDT spt Binance/Bybit (`"BTC/USDT:USDT"`) — krn `--symbols` di CLI dipakai bareng utk semua `--venues` dalam satu invocation, Hyperliquid **wajib dijalankan di invocation terpisah** dgn symbol list-nya sendiri (lihat contoh command di bawah), gak bisa dicampur `--venues binance bybit hyperliquid` pakai satu `--symbols` yg sama.

Auto-provision `venue`/`instrument` row di first-run (idempotent per venue, sama pola dgn `platform_user` auto-provision di `api-gateway/deps.py`) — `venue.venue_type` sekarang per-venue dari `VENUES` dict (`"cex"` utk Binance/Bybit, `"dex"` utk Hyperliquid, sesuai `ck_venue_type` constraint di DB), upsert `funding_rate`/`ohlcv` via `db.merge()` (re-run aman, gak duplikat instrument/venue — funding_rate/ohlcv sendiri emang nambah row baru tiap fetch selama `ts` beda, itu time-series by design bukan bug), dan tulis `data_source_health` per venue+data_type (sukses/gagal + `consecutive_failures`) tiap kali fetch.

Standalone script utk sekarang, **belum di-wire ke Inngest** (self-hosted job scheduler yg direncanakan di B.1/B.9 — belum ada infra-nya sama sekali di repo ini). Jalankan manual/cron sementara.

**Catatan verifikasi**: logic upsert/idempotency/health-tracking (termasuk multi-venue: venue row terpisah per venue_type, instrument per-venue, health per venue+data_type), fallback `fetchFundingRates`, parsing `interval` dinamis, dan skip kredensial utk `api_key_env=None` — semua diverifikasi lewat mocked exchange object + Postgres 16 lokal beneran (migrasi `alembic upgrade head` dijalankan penuh, bukan cuma import check), termasuk regression check eksplisit bhw Binance/Bybit tetap dapat `funding_interval_hours=8`/`venue_type=cex` spt sebelumnya (dari sandbox Claude Code, yg diblokir network policy-nya ke `fapi.binance.com`/Bybit/`api.hyperliquid.xyz` — dikonfirmasi via `ex.load_markets()` beneran timeout `NetworkError`, bukan diasumsikan). **Panggilan jaringan asli ke Binance + Neon production sudah dites user sendiri & BERHASIL** (`BTC/USDT:USDT` & `ETH/USDT:USDT` di Binance, funding_rate + ohlcv, lewat proxy Webshare.io) — lihat gotcha proxy di bawah kalau nemu `InvalidProxySettings`/`407 Proxy Authentication Required` pas setup pertama kali.

**Hyperliquid sekarang JUGA sudah TERVERIFIKASI via jaringan asli** (user jalankan dari Termux/Android, bukan proxy cloud — Hyperliquid gak butuh proxy sama sekali beda dari Binance) lewat `ccxt.hyperliquid()` langsung ke `api.hyperliquid.xyz`: `load_markets()` sukses (752 market), `has.get("fetchFundingRate")` konfirmasi `False` sesuai dugaan (fallback plural yg beneran kepakai), `fetch_funding_rates(["BTC/USDC:USDC"])` balikin data asli dgn `interval: "1h"` (bukan 8h — bug lama kalau gak di-fix bakal salah total di sini) dan `fetch_ohlcv` balikin candle asli. Setup Termux sempat kena 2 gotcha nyata: (1) `pip install ccxt` gagal build `cryptography` dari source krn rustup gak support target `aarch64-unknown-linux-android` — fix pakai `pkg install python-cryptography` (build Termux sendiri, hindari compile Rust manual); (2) `coincurve` (dependency ccxt yg dideklarasikan wajib di metadata, padahal cuma dipakai opsional via `try/except ImportError` di kode ccxt utk ECDSA signing yg gak pernah kepakai script publik-data ini) bikin build macet lama (butuh toolchain autotools+libsecp256k1) — fix pakai `pip install ccxt --no-deps` lalu `pip install requests certifi typing_extensions` manual (`aiohttp`/`aiodns`/`yarl` versi async jg gak kepakai di sini, script ini sync-only). **Bybit sekarang JUGA sudah TERVERIFIKASI via jaringan asli** (3 Juli 2026, lewat deploy `ingestion-worker` di Railway, PROXY_URL yg sama dgn Binance) — backfill 365 hari x BTC/USDT+ETH/USDT sukses (8760 candle per instrumen, sama persis hasilnya dgn Binance di deploy yg sama), lanjut ke live poll (`funding_rate`+`ohlcv` OK) tanpa error apapun. Ini verifikasi jaringan asli PERTAMA utk Bybit di seluruh riwayat proyek ini (sebelumnya cuma mocked) — jalan sukses dari percobaan pertama, gak perlu fix apapun.

**Fallback chain (`connectors/cex/native_fallback.py`) — Fase 1 B.9/B.11 gap, sekarang ada**, scope sengaja sempit sesuai keputusan: retry/fallback utk `funding_rate`+`ohlcv` doang (bukan ingestion `liquidation_event` baru, itu beda kerjaan krn primary-nya sendiri blm ditulis). ccxt SELALU dicoba dulu tiap run (supaya path yg udah pulih otomatis kedeteksi lagi) — cuma kalau venue+data_type itu udah gagal `FALLBACK_THRESHOLD=3` kali berturut-turut (dibaca dari `data_source_health.consecutive_failures` yg emang udah ada dari awal, gak perlu migration baru), kegagalan attempt berikutnya jg nyoba native REST exchange itu sendiri sblm nyerah — `ingest_funding_rate()`/`ingest_ohlcv()` di `ingest.py` sekarang balikin string sumber data (`"ccxt"` atau `"native fallback"`) buat logging, dan `data_source_health` cuma dianggap gagal kalau KEDUA jalur gagal.

Native fallback cuma ada utk **Binance + Bybit** (2 venue yg disebut eksplisit di B.11), bukan Hyperliquid — connector ccxt Hyperliquid udah manggil `api.hyperliquid.xyz` langsung tanpa layer SDK/vendor terpisah (ketauan pas nambah connector Hyperliquid kemarin), jadi gak ada "native path" lain yg beda buat di-fallback-in. `native_fallback.FALLBACKS.get(venue_name, {}).get(data_type)` balikin `None` utk Hyperliquid — sama persis kayak sblm fallback chain ada, gak ada perubahan behavior.

Native fallback ini nolongin dari bug/regresi/outage di level **library ccxt**, BUKAN dari outage exchange itu sendiri — dua-duanya (ccxt & native fallback) tetap nembak server exchange yg sama, jadi kalau Binance/Bybit beneran down, kedua jalur bakal gagal bareng.

`native_fallback.py` sekarang JUGA baca `PROXY_URL` (sama pola kayak `ccxt_generic.py`, lihat `_proxies()`) — **bug nyata ketemu pas verifikasi jaringan asli** (bukan diasumsikan dari awal): tanpa ini, native fallback nembak endpoint yg sama persis dgn ccxt tanpa proxy, jadi kalau alasan ccxt gagal itu IP-level blocking (persis skenario yg PROXY_URL emang didesain buat nyelesain), native fallback bakal ikutan gagal krn ke-block di jalur network yg sama — bikin seluruh fitur fallback jadi gak berguna justru di skenario yg seharusnya dia tangani.

**Catatan verifikasi fallback chain**: diverifikasi end-to-end via mocked ccxt exchange object + Postgres 16 lokal beneran — kegagalan ke-1/2/3 (di bawah threshold) TIDAK trigger fallback, kegagalan ke-4 (udah >= threshold) trigger fallback & berhasil (data yg ke-tulis ke DB cocok data fallback, bukan data ccxt yg gagal, `consecutive_failures` balik ke 0), skenario "ccxt DAN fallback dua-duanya gagal" tetap gagal bersih (gak ada row nyasar ke-tulis, `consecutive_failures` tetap naik), skenario Hyperliquid (gak py fallback terdaftar) gak crash sama sekali (perilaku identik sblm fallback chain ada), dan `_proxies()` — kosong kalau `PROXY_URL` gak di-set, keisi & kepakai bener di tiap `requests.get()` kalau di-set (diverifikasi via monkeypatch `requests.get`, bukan diasumsikan). Satu temuan gak sengaja pas testing awal: skenario ohlcv-tanpa-mock-fallback beneran manggil `fapi.binance.com/fapi/v1/klines` sungguhan dari sandbox ini dan dapet `451` (diblokir).

**Percobaan verifikasi jaringan asli dari Termux (Android/Pixel 8, jaringan seluler Indonesia)**: `fapi.binance.com` GAGAL — bukan soal certifi/Termux/Python, dibuktikan `curl` polos (tanpa Python sama sekali) juga gagal persis sama: `SSL: no alternative certificate subject name matches target hostname 'fapi.binance.com'` (hostname mismatch, bukan "unable to get local issuer certificate" spt awalnya keliatan sblm didiagnosa lebih jauh). Ini indikasi kuat interception di level jaringan (ISP/operator seluler nyuguhin sertifikat yg salah, bukan nolak koneksi bersih) — kategori masalah yg sama persis kenapa `PROXY_URL`/Webshare dulu dibutuhkan buat jalur ccxt dari Railway, cuma manifestasinya beda (cert mismatch di jaringan seluler vs `451` bersih dari IP datacenter). **Field-shape response Binance `premiumIndex`/`klines` & Bybit `v5/market/tickers`/`v5/market/kline` di `native_fallback.py` masih didasarkan dokumentasi API publik, belum kebuktiin lewat panggilan asli** — nunggu percobaan lain (proxy/VPN, atau dari jaringan yg gak ke-block) sblm dianggap kelar verifikasi jaringannya, tapi bug `PROXY_URL` di atas sudah nyata ketemu & fixed dari usaha ini walau endpoint-nya sendiri blm berhasil ditembus.

**Backfill historis + worker polling kontinu (`worker.py`, `ccxt_generic.fetch_ohlcv_range`, `ingest.backfill_ohlcv`/`needs_backfill`) — 3 Juli 2026**, jawaban langsung ke pertanyaan founder "gimana narik histori candle lebih panjang secara efisien, sambil tetap jalan live" (dan "apa CoinGlass Hobby bisa bantu" — jawabannya TIDAK: OHLCV di sistem ini emang gak pernah dari CoinGlass sama sekali, cuma dari CCXT/exchange langsung spt di atas, dan Binance sendiri kasih histori OHLCV penuh gratis lewat public API tanpa batas plan — masalahnya emang di `ingest.py`/scheduling, bukan data vendor).

- `ccxt_generic.fetch_ohlcv_range()` (baru): paging maju dari `since_ms` ke `until_ms` (default sekarang), advance cursor pakai `exchange.parse_timeframe(timeframe)` (reuse ccxt, bukan reimplementasi), berhenti kalau exchange balikin page kosong (histori exchange abis) atau udah lewatin `until_ms`. `make_exchange()` sekarang set `enableRateLimit=True` (sebelumnya GAK di-set sama sekali) — ccxt jadi otomatis nge-throttle tiap call sesuai limit exchange itu sendiri, itulah yg bikin paging cepet TANPA kena 429 (diverifikasi: 90 hari candle 1h = 2160 baris kepaging cuma ~2.5 detik lewat Hyperliquid asli).
- `ingest.needs_backfill()` (pure function) + `ingest.backfill_if_needed()`: skip backfill kalau `ohlcv` udah punya data yg cover range yg diminta — bikin worker aman di-restart kapan aja tanpa nge-ulang paging yg udah pernah jalan.
- **Bug nyata ketemu pas testing manual end-to-end thd Postgres 16 lokal + Hyperliquid asli** (bukan cuma lolos test sintetis): `needs_backfill()` versi pertama bandingin `existing_earliest > requested_since` mentah-mentah — gagal skip di run ke-2, krn candle nempel di boundary timeframe (mis. diminta "since 07:14", tapi candle 1h pertama yg beneran ada itu jam 08:00 — LEBIH LAMBAT dari 07:14 walau sebenernya udah ke-cover penuh, gak ada candle 1h antara 07:14-08:00 yg kelewat). Fix: `needs_backfill()` sekarang terima `tolerance` (satu durasi timeframe), `backfill_if_needed()` ngitung dari `exchange.parse_timeframe()`. Ke-detect krn dites REAL (backfill 5 hari, run ulang, cek log — harusnya "SKIPPED" tapi malah re-backfill 120 candle lagi), bukan diasumsikan bener dari logic doang.
- `worker.py` (baru): backfill sekali di awal (skip otomatis kalau udah cover, jadi murah di-restart kapan aja) → loop polling selaras ke penutupan candle tiap timeframe (`seconds_until_next_close()`, pure function, ditest terpisah dari real-time sleep) → ulang. Backfill jalan SYNCHRONOUS sebelum loop poll (bukan thread terpisah) — sengaja, krn `enableRateLimit` udah bikin paging cepet banget (detik-menit, bukan jam), gak worth kompleksitas+resiko sharing DB session/exchange object across thread.
- **Diverifikasi end-to-end penuh**: Postgres 16 lokal (migrasi 0001-0005 penuh) + Hyperliquid asli (BUKAN mocked) — backfill 5 hari (120 candle exact), re-run correctly SKIP, minta lebih dalam (10 hari) correctly extend jadi 240 candle tanpa duplikat, `worker.py` full cycle (backfill→poll→hitung sleep→shutdown bersih via `should_stop`) semua jalan bener. **Binance GAK bisa dites jaringan asli dari sandbox Claude Code round ini** — bukan proxy/cert issue (awalnya keliatan begitu dari `SSLCertVerificationError`), ternyata `curl` polos ke `fapi.binance.com/fapi/v1/time` balikin pesan block eksplisit dari Binance sendiri: *"Service unavailable from a restricted location according to 'b. Eligibility'..."* — beda dgn `451` yg pernah ketemu sebelumnya (di atas), tapi kesimpulannya sama: exchange venue itu sendiri yg nge-block, konsisten sama semua temuan blocking sebelumnya di file ini.
- 21 test baru (`tests/test_ccxt_generic.py`, `tests/test_worker.py`, `tests/test_ingest.py`) — semua pure-logic (fake ccxt exchange object, gak ada network/DB beneran), dijalankan di CI job baru `ingestion-test` (sebelumnya `apps/products/trading/ingestion` gak ke-cover CI sama sekali).
- **Railway**: worker ini dijalanin sbg SERVICE TERPISAH (`railway.ingestion-worker.toml` di repo root). **SUDAH DIVERIFIKASI thd Railway asli (3 Juli 2026)** — deploy pertama langsung sukses tanpa perlu fix apapun: Deploy Logs nunjukin urutan lengkap sesuai desain (backfill 365 hari x 4 kombinasi venue/simbol, 8760 candle masing2, lanjut `backfill done -- entering live poll loop`, 1 siklus poll funding_rate+ohlcv per instrumen, `sleeping 2718s until next 1h close`). Total **35.040 candle baru** masuk production dari deploy ini doang. Lihat komentar di file itu + `docs/deployment-runbook.md` gotcha #8 utk detail lengkap kenapa deps-nya digabung ke `requirements.txt` root yg sama, bukan file terpisah.

## Local dev

```bash
cd apps/products/trading/ingestion
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
export DATABASE_URL="postgresql://<user>:<pass>@<host>/<db>"   # +psycopg dipaksa otomatis, lihat kinetiq_db.engine
# Opsional -- lihat .env.example utk penjelasan lengkap:
# export BINANCE_API_KEY="..."       # HARUS read-only key, funding_rate/ohlcv itu public endpoint
# export BINANCE_API_SECRET="..."
# export BYBIT_API_KEY="..."         # sama, harus read-only
# export BYBIT_API_SECRET="..."
# export PROXY_URL="http://user:pass@proxy-host:port"   # mis. Webshare.io, dipakai semua venue
#                                     # (ganti nama dari BINANCE_PROXY_URL versi sebelumnya)
PYTHONPATH=../../../../packages/db/src python ingest.py --venues binance bybit --symbols "BTC/USDT:USDT" "ETH/USDT:USDT" --timeframe 1h --limit 100

# Hyperliquid: invocation terpisah krn quote currency beda (USDC, bukan USDT) --
# gak bisa dicampur ke command di atas, --symbols dipakai bareng utk semua --venues.
# Public endpoint, gak butuh API key sama sekali (walletAddress/privateKey cuma
# perlu utk order/transfer, gak dipakai script ini).
PYTHONPATH=../../../../packages/db/src python ingest.py --venues hyperliquid --symbols "BTC/USDC:USDC" "ETH/USDC:USDC" --timeframe 1h --limit 100

# Backfill histori (satu kali, aman diulang -- skip otomatis kalau udah cover):
PYTHONPATH=../../../../packages/db/src python ingest.py --venues binance bybit --symbols "BTC/USDT:USDT" "ETH/USDT:USDT" --timeframe 1h --backfill-since-days 365

# Worker kontinu (backfill sekali di startup, lalu poll selaras tiap penutupan candle):
PYTHONPATH=../../../../packages/db/src python worker.py --venues binance bybit --symbols "BTC/USDT:USDT" "ETH/USDT:USDT" --timeframe 1h --backfill-since-days 365
```

`BINANCE_API_KEY`/`BYBIT_API_KEY`/`PROXY_URL` menyelesaikan dua masalah yg beda: API key (harus read-only per venue, jangan pernah kasih izin trading/withdrawal — script ini gak pernah submit order) cuma soal rate limit; proxy soal IP yg mungkin di-block/dibatasi (mis. IP cloud/datacenter kayak Railway) — request yg udah authenticated dari IP yg di-block tetap ke-block, jadi API key doang gak nyelesain masalah blocking. `PROXY_URL` sengaja satu env var dipakai semua venue (bukan per-exchange spt API key), krn IP blocking itu masalah jaringan, bukan spesifik ke satu exchange.

**Gotcha proxy yg kejadian pas setup real pertama kali** (keduanya sudah fixed di kode/didokumentasikan di sini biar gak keulang):
- `ccxt.base.errors.InvalidProxySettings: ...multiple conflicting proxy settings...` — sudah di-fix di `ccxt_generic.py` (cuma set `httpsProxy`, jangan `httpProxy` bareng, krn semua venue di sini selalu `https://`).
- `407 Proxy Authentication Required` — ini BUKAN soal concurrent user/plan Webshare, murni salah copy username/password proxy (`user:pass@host:port`). Test isolasi paling gampang, gak perlu Python sama sekali:
  ```bash
  curl.exe --proxy "http://user:pass@host:port/" https://ipv4.webshare.io/
  ```
  Kalau ini balikin sebuah IP address, kredensial proxy-nya valid dan masalahnya ada di tempat lain (mis. `DATABASE_URL` yg salah, bukan proxy).
